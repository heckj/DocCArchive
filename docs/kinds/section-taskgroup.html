1. How swift-docc-render consumes this

A grep of src/components/ and src/views/ for the literal string "taskGroup" finds nothing — this kind never appears as a wire-level discriminator that swift-docc-render switches on. Instead, a grep for topicSections (the RenderNode field that holds an array of taskGroup objects) turns up exactly two consuming files: DocumentationTopic.vue#L137-L143 and the shared Topics.vue/TopicsTable.vue pair it delegates to. The same shape is reused, unconditionally, for two sibling RenderNode fields — defaultImplementationsSections and seeAlsoSections (DocumentationTopic.vue#L144-L160) — because all three RenderNode fields are typed as an array of the identical title/abstract/discussion/identifiers/generated/anchor object; swift-docc's own name for that payload type is TaskGroupRenderSection, and its wire form has no kind discriminator field at all (see Section 3).

DocumentationTopic.vue (the top-level component for API/symbol reference pages) passes topicSections straight into <Topics>: 

<Topics
  v-if="shouldRenderTopicSection"
  :sections="topicSections"
  :isSymbolDeprecated="isSymbolDeprecated"
  :isSymbolBeta="isSymbolBeta"
  :topicStyle="topicSectionsStyle"
/>

(DocumentationTopic.vue#L137-L143.) DocumentationTopic/Topics.vue#L11-L20 is a thin wrapper that just forwards sections/topicStyle straight through to <TopicsTable>, which is where each individual taskGroup entry is actually unpacked and rendered: 

<ContentTableSection
  v-for="(section, i) in sectionsWithTopics"
  :key="`${section.title}_${i}`"
  :title="section.title"
  :anchor="section.anchor"
  :class="{ 'no-title': !section.title }"
>
  ...
  <template v-if="section.abstract" #abstract>
    <ContentNode :content="section.abstract" />
  </template>
  <template v-if="section.discussion" #discussion>
    <ContentNode :content="section.discussion.content" />
  </template>
  <template v-if="shouldRenderList">
    <TopicsLinkBlock v-for="topic in section.topics" ... />
  </template>
  <TopicsLinkCardGrid v-else :items="section.topics" :topicStyle="topicStyle" class="topic" />
</ContentTableSection>

(DocumentationTopic/TopicsTable.vue#L11-L51.) The computed property sectionsWithTopics is what resolves each taskGroup object's raw identifiers array of topic-reference URLs into full reference objects, by looking each one up in the page-level references map (a mixin, referencesProvider): 

sectionsWithTopics() {
  return this.sections.map(section => ({
    ...section,
    topics: section.identifiers.reduce(
      (list, id) => (this.references[id] ? list.concat(this.references[id]) : list),
      [],
    ),
  }));
},

(TopicsTable.vue#L110-L118.) Depending on topicStyle (a sibling RenderNode field, topicSectionsStyle, one of list/compactGrid/detailedGrid/hidden), each resolved topic renders either as a TopicsLinkBlock row (TopicsLinkBlock.vue#L11-L27) or as a card inside TopicsLinkCardGrid. The outer chrome (section title, top border, spacing) comes from ContentTable.vue (ContentTable.vue#L11-L18) and ContentTableSection.vue (ContentTableSection.vue#L11-L29), one ContentTable per RenderNode field and one ContentTableSection per taskGroup entry within it.

The exact same TopicsTable.vue component (minus the list/grid style toggle) is reused verbatim by SeeAlso.vue (SeeAlso.vue#L11-L19, title "See Also") and by DefaultImplementations.vue (DefaultImplementations.vue#L11-L19, title "Default Implementations") — confirming that from swift-docc-render's point of view, "Topics", "See Also", and "Default Implementations" are three independent RenderNode array fields holding the identical taskGroup item shape, not three different render kinds.

See also: The references map — how this and every other reference-dictionary lookup works across kinds.

3. Data sheet

Source: Sources/DocCArchive/Vendored/RenderNode.spec.json, schema object #/components/schemas/TasksGroupSection (note the schema's own name differs from both the wire discriminator string "taskGroup" used in swift-docc's RenderSectionKind enum and from swift-docc's Swift type name TaskGroupRenderSection — see the naming note below the table), cross-checked against Components.Schemas.TasksGroupSection in Sources/DocCArchive/generated/Types.swift (lines 6301–6347). This schema is referenced from three sibling fields on DocumentationRenderNode: topicSections, seeAlsoSections, and defaultImplementationsSections (spec lines 2111–2128; generated lines 5367, 5369, 5371).

FieldTypeRequiredEnum values / defaultNote
kind — not present on the wire — Unlike most RenderSectionKind cases, TaskGroupRenderSection declares public let kind: RenderSectionKind = .taskGroup in Swift but its CodingKeys enum (swift-docc, TaskGroupRenderSection.swift lines 60–62) deliberately omits kind, so the JSON never carries a "kind": "taskGroup" field. The vendored TasksGroupSection schema matches this: it has no kind property at all. A consumer can only tell which RenderNode field a given taskGroup object came from by which array (topicSections / seeAlsoSections / defaultImplementationsSections) it was found in.
title string no — The task group's heading, e.g. "Classes" or an author-written ## Getting Started heading. Absent for the single unnamed catch-all group DocC sometimes emits.
abstract array of RenderInlineContent no — The first paragraph of authored prose under the group's heading, if any. Never populated for automatically-curated groups (generated: true).
discussion ContentRenderSection (vendored schema) / any RenderSection (swift-docc, in practice a ContentRenderSection) no — Additional authored prose after the abstract and before the link list, wrapped in its own nested render-section object (accessed in swift-docc-render as section.discussion.content).
identifiers array of string yes — Topic-graph reference identifiers (URLs in the doc:// scheme, or resolved external URLs) for the symbols/articles curated into this group. swift-docc-render resolves each one against the RenderNode's top-level references map before display.
generated boolean no (generated schema field is optional; swift-docc always emits it as false when omitted) default false Encoded only when true (swift-docc's encode(to:) uses if generated { try container.encode(...) }) — true marks a group DocC synthesized automatically (e.g. grouping all classes together) rather than one an author wrote under a ## Topics heading.
anchor string no defaults to a URL-readable slug of title Used as the deep-link fragment for the group's heading; swift-docc computes it via title.map(urlReadableFragment) when not explicitly supplied.

4. HTML/CSS mapping in DocC-Render output

Rendered DOM, outer to inner, for the "Topics" content table (identical structure for "See Also" and "Default Implementations", just with a different outer title/anchor):

  • section.contenttable.alt-light — the root element of ContentTable.vue (ContentTable.vue#L11-L18), one per RenderNode field (topicSections/seeAlsoSections/defaultImplementationsSections). Its scoped styles set --section-spacing-single-side from $section-spacing-single-side (40px, src/styles/core/_vars.scss line 61) and, for the non-minimized case, apply the dynamic-content-container mixin (src/styles/core/_breakpoints.scss line 405) to size the content column responsively.
  • div.container — wraps the heading and the per-group sections; gains .minimized-container (tighter, 1.5em spacing) when the page is rendered in the IDE-embedded "minimized" mode.
  • div.contenttable-section[.no-title] — one per taskGroup entry, the root element of ContentTableSection.vue (ContentTableSection.vue#L11-L29). Padded via $contenttable-spacing-single-side (2.353rem, src/styles/core/_vars.scss line 26); gets the no-title modifier class when the group's title is absent, which suppresses the first item's top margin.
  • h3.contenttable-title — the group's title, rendered by LinkableHeading and anchored with section.anchor (or an authored <h3> via the #title slot when wrapTitle is set, as DefaultImplementations.vue does).
  • div.section-content > .content — wraps the resolved abstract and discussion prose (each passed through ContentNode.vue) and the topic list/grid itself; TopicsTable.vue's own scoped style block gives .topic items margin-top: 15px, collapsed to 0 on the first child when the section has no title (TopicsTable.vue#L123-L135).
  • div.link-block (list style) or a card inside .TopicsLinkCardGrid (grid styles) — one per resolved topic in identifiers, from TopicsLinkBlock.vue (TopicsLinkBlock.vue#L11-L27) or TopicsLinkCardGrid.vue depending on topicStyle.

SCSS files involved: ContentTable.vue, ContentTableSection.vue, and TopicsTable.vue each carry their own <style scoped lang="scss"> block, all importing the shared src/styles/_core.scss for the breakpoint() mixin and the spacing variables noted above. CSS custom properties consumed include --color-grid (the content-table's top border color, set on ContentTable.vue's .title) and --content-table-title-border-width (theme override for that border's width, default 1px).

DocCArchive transform status: not yet implemented. MyHTMLTransformer and MyMarkdownTransformer in Sources/DocCArchive/DocCArchive.swift are still stub placeholders — the HTML transformer returns a hardcoded comment regardless of input and the Markdown transformer returns an empty string — so there is no code path today that walks a RenderNode's topicSections/seeAlsoSections/defaultImplementationsSections arrays into a "Topics" heading plus a link list in either output format.

5. Modern CSS-only alternative

NOT APPLICABLE

Current behavior in play: rendering a taskGroup entry is a one-time, non-interactive template computation. TopicsTable.vue's sectionsWithTopics computed property (TopicsTable.vue#L110-L118) runs once per render pass to resolve identifiers against the page's reference map, and the v-for loops over groups and topics are ordinary static list rendering — there is no component-local mutable state being toggled, no click handler, and no later state transition (unlike, say, a tab navigator whose visible pane changes after mount). The only "dynamic" aspect — shouldRenderList switching between the TopicsLinkBlock list layout and the TopicsLinkCardGrid card layout — is resolved once from the RenderNode's own topicSectionsStyle field at render time, not from user interaction. Because there is no interactive/stateful behavior here to replace, none of <details>/<summary>, radio+:checked, :target, the Popover API, <dialog>, :has(), container queries, CSS anchor positioning, or view-transitions has anything to hold on this kind's behalf; a static server-rendered HTML document with the same nested-list/heading structure would be behaviorally identical. (The per-group deep-link anchor, e.g. #Classes, is already just a plain HTML id attribute plus a same-page fragment link — ordinary browser navigation, not JS-held state.)
