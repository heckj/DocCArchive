1. How swift-docc-render consumes this

article is one of two possible values of the top-level kind discriminator on a DocumentationRenderNode (the other is "symbol"). It marks a standalone, free-form documentation page — an "extension article" or top-level collection article written in a .md file in a DocC catalog — as opposed to a page that documents a specific symbol/declaration. Both share the exact same JSON schema and Vue rendering path; the difference is only in which optional fields are populated (an article has no primaryContentSections of kind declarations, for example).

Do not confuse this with views/Topic.vue's TopicKind.article. swift-docc-render's route table (src/routes.js) sends any path starting with /documentation to views/DocumentationTopic.vue, which is the actual consumer of a DocumentationRenderNode — this is the kind described on this page. Paths starting with /tutorials instead go to views/Topic.vue, whose own, differently-scoped TopicKind.article constant (src/views/Topic.vue#L37-L40) dispatches to components/Article.vue, the renderer for tutorial "article" chapters (hero/callToAction/assessments sections) that belong to a TutorialsPageRenderNode, a completely different top-level schema variant. The two "articles" only share a wire-value string, not a schema.

views/DocumentationTopic.vue wraps a single child, DocumentationLayout, whose #content slot renders components/DocumentationTopic.vue (aliased as Topic in the view). This component takes the node's fields as flattened props (role, roleHeading, title, abstract, topicSections, etc. — see props.role) rather than switching on kind directly. It renders unconditionally for both kind: "symbol" and kind: "article"; sections that don't apply to a plain article (declarations, relationships, default implementations) are simply absent from the JSON and so nothing renders for them — there is no v-if="kind === 'article'" branch.

The one place the render code visibly reacts to an article is via the finer-grained metadata.role field (DocumentationRole enum, which also includes "article" as one of its values, distinct from but usually aligned with the top-level kind). role is passed straight through to DocumentationHero.vue, which uses it to pick the hero's icon and accent color: 

// src/constants/TopicTypes.js
export const TopicTypes = { article: 'article', /* … */ };

// src/constants/HeroColors.js
export const HeroColorsMap = {
  [TopicTypes.article]: HeroColors.teal,
  /* … */
};

// src/components/TopicTypeIcon.vue
import ArticleIcon from 'theme/components/Icons/ArticleIcon.vue';
// … [TopicTypes.article]: ArticleIcon, …

so an article page gets a teal-tinted hero background and the dedicated ArticleIcon watermark, in contrast to a symbol page's kind-specific color (purple for struct, blue for method, etc). metadata.roleHeading supplies the literal eyebrow text above the title — for articles this is the plain string "Article", set on the swift-docc side (see step 3 below), and swift-docc-render simply displays it verbatim via Title's eyebrow prop.

On the swift-docc authoring side, a plain Markdown file in a DocC catalog (no @Tutorial or @TutorialTableOfContents directive) is modeled as an Article semantic node and translated by RenderNodeTranslator.visitArticle (Sources/SwiftDocC/Model/Rendering/RenderNodeTranslator.swift:624-625), which constructs RenderNode(identifier: identifier, kind: .article) and then sets node.metadata.role/roleHeading (lines 763-771, the latter hard-coding the string "Article") — this is the origin of the JSON in step 2 below.

In this repository (DocCArchive), a parsed article node would arrive as the .DocumentationRenderNode case of the generated RenderNode enum (Components.Schemas.RenderNode, Sources/DocCArchive/generated/Types.swift:2883-2916) with kind == .article. Turning it into HTML or Markdown output is not yet implemented: MyHTMLTransformer and MyMarkdownTransformer (Sources/DocCArchive/DocCArchive.swift:242-254) are still stub placeholders that ignore the node's contents and return a hard-coded HTML comment / an empty string, respectively, regardless of whether the node is a symbol or an article.

4. HTML/CSS mapping in DocC-Render output

Root wrapper: <div class="doc-topic"> (src/components/DocumentationTopic.vue:13, styled in the component's own <style scoped lang="scss"> block starting at line 752/796), shared verbatim by symbol and article pages alike. Inside it:

  • <div class="documentation-hero"> / documentation-hero__content / documentation-hero__above-content — from src/components/DocumentationTopic/Hero/DocumentationHero.vue; its scoped SCSS defines the gradient background via the CSS custom properties --accent-color and --standard-accent-color, which are set inline per-instance from the resolved HeroColorsMap entry for role: "article" (teal) and read by --color-documentation-intro-fill/--color-documentation-intro-accent theme variables as fallbacks.
  • <div class="icon"> containing the ArticleIcon SVG watermark (background-icon class), opacity/blend-mode controlled by $doc-hero-icon-opacity/$doc-hero-icon-effect SCSS variables in the same file.
  • The page title block uses the shared Title component with an eyebrow prop set to metadata.roleHeading ("Article").
  • Body content (from primaryContentSections) is rendered through PrimaryContent/ContentNode.vue — the same block-content dispatch table used by every other block-content-bearing kind — so an article's headings, paragraphs, code listings, etc. get identical DOM/CSS to the corresponding content in a symbol page.

No article-specific SCSS file exists; all of the above is either inherited from the shared DocumentationTopic.vue/DocumentationHero.vue styles or parameterized by the role-driven color/icon lookup tables in src/constants/TopicTypes.js and src/constants/HeroColors.js.

5. Modern CSS-only alternative

Not applicable

Current behavior: rendering an article-kind DocumentationRenderNode is a one-shot server/build-time-to-DOM mapping with no client-side interactive state to hold. Vue computes role/roleHeading/hero color from static JSON props once, on mount, and there is no toggle, tab, disclosure, or other stateful widget anywhere in this rendering path (contrast with something like a tab navigator, which holds currentTitle in component data and swaps visible panels via v-show). Any interactivity a reader encounters on an article page (a code-listing tab switcher, a disclosure triangle in a "Topics" grid, etc.) belongs to a nested block/inline content kind, not to the article node kind itself — those child kinds should be judged on their own reference pages. There is therefore no single native HTML/CSS mechanism to "replace" here: the node kind itself is a static content container, and a CSS-only alternative isn't a meaningful question for it.
