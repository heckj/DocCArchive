1. How swift-docc-render consumes this

"topic" is the type discriminator used on entries in a RenderNode's top-level references dictionary that point at another page of documentation (as opposed to an image, a download file, an external link, etc.). Swift-docc's TopicRenderReference is the Swift model for this shape; on the wire it is keyed by its identifier (a doc:// URL) inside RenderNode.references, and other parts of the render node — most commonly a "Topics" section's identifiers list — reference it by that identifier instead of inlining the full object repeatedly.

The resolution and rendering chain in swift-docc-render is:

  1. referencesProvider (a Vue mixin) exposes the render node's whole references map as a computed property, backed by the app's Vuex-like store.state.references: src/mixins/referencesProvider.js#L11-L27.
  2. TopicsTable.vue (the component behind a page's rendered "Topics" section) takes each section.identifiers entry and looks it up in that map (this.references[id]), building a section.topics array of resolved TopicRenderReference objects: TopicsTable.vue#L108-L118. It then either hands each resolved topic to TopicsLinkBlock (list style, the default) or the whole array to TopicsLinkCardGrid (grid/card style), switching on a topicStyle prop: TopicsTable.vue#L35-L50.
  3. TopicsLinkBlock.vue is the component that actually renders one topic reference as a link row. Its root element is <div class="link-block">, containing a <router-link> (internal) or <a> (when topic.type === 'link') wrapping a TopicLinkBlockIcon (role glyph), the title (as DecoratedTopicTitle if the reference carries symbol fragments, else plain text), and a following .abstract block with the topic's abstract content, badges for beta/deprecated status, and any tags: TopicsLinkBlock.vue#L11-L57. The component's own prop validator (not just documentation) enumerates the exact shape it expects a resolved topic reference to satisfy: TopicsLinkBlock.vue#L104-L119.
  4. Separately, the generic Reference.vue component (used for inline cross-references inside prose, not just the "Topics" section) accepts a kind prop whose valid values are documented in its header comment as "See DocumentationRenderNode.kind"/TopicRenderReference.type in the JSON schema, and uses it (together with role) to decide whether to render a symbol-styled link (ReferenceInternalSymbol/ReferenceExternalSymbol) or a plain link (ReferenceInternal/ReferenceExternal): Reference.vue#L40-L96.

The role-to-icon mapping used by TopicLinkBlockIcon (shown next to each topic link, driven by the reference's role field, e.g. symbol, article, collection) lives at TopicLinkBlockIcon.vue#L17-L47.

See also: The references map — how this and every other reference-dictionary lookup works across kinds.

4. HTML/CSS mapping in DocC-Render output

  • div.link-block — root element rendered by TopicsLinkBlock.vue for each resolved topic; modifier classes include has-inline-element (no abstract/badges to show below the title) and a "changes" class when API-diff highlighting is active.
  • a.link / router-link.link — the actual clickable link wrapping the icon + title; gets .deprecated (strikethrough) and .has-adjacent-elements when an abstract/badges follow.
  • div.topic-icon-wrapper > role-specific SVG icon component (from TopicLinkBlockIcon.vue).
  • div.abstract — wraps the rendered abstract ContentNode, plus div.topic-keyinfo for plist "Key/Name" lines and RequirementMetadata ("Required." / default-implementation count) for protocol requirements.
  • span.badge (component Badge.vue) — used for the beta badge, deprecated badge, and the first entry of tags.
  • Enclosing structure from TopicsTable.vue: ContentTable > ContentTableSection > (div.topic wrapper around each TopicsLinkBlock, or a TopicsLinkCardGrid when the page uses card/grid style instead of the list style).

SCSS: the scoped <style> block inside TopicsLinkBlock.vue defines .link-block, .link, .abstract, .topic-required, and .deprecated rules and imports the shared docc-render/styles/_core.scss. It consumes two SCSS variables defined centrally in src/styles/core/_vars.scss: $topic-link-icon-width (1.294rem) and $topic-link-icon-spacing (1rem), used to indent the abstract/badges so they line up with the title text after the icon. The scoped style block inside TopicsTable.vue separately sets the vertical spacing (margin-top: 15px) between successive .topic entries.

5. Modern CSS-only alternative

Not applicable

A resolved topic reference, as rendered by TopicsLinkBlock.vue, is not an interactive/stateful widget: there is no client-side toggle, tab, disclosure, or modal state being held in Vue component data for a single topic link row. It is a link (an <a> or Vue-Router <router-link>) with a decorated label, an optional abstract, and optional status badges — all computed once from static props (topic.title, topic.abstract, topic.beta, topic.deprecated, etc.) and rendered declaratively; nothing in TopicsLinkBlock.vue's computed block reacts to user interaction (clicking simply navigates, which is native browser/router behavior, not a piece of UI state a CSS mechanism like :checked or :target would need to replace). The one piece of "dynamism" — API-diff highlighting via changesClasses — is itself just a class computed from external route/query state (which diff view is selected), not something toggled by user interaction with this component. There is therefore no equivalent click-driven show/hide, tab-switch, or expand/collapse behavior for a native HTML mechanism (<details>, radio+:checked, :target, Popover API, <dialog>) to take over; this kind is a static, link-based rendering.
