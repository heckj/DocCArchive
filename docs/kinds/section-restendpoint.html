1. How swift-docc-render consumes this

restEndpoint is one of the section kinds a REST-symbol page's primaryContentSections array can contain (alongside restParameters, restHeaders, restCookies, restBody, restResponses, and the ordinary symbol sections). The dispatch table lives in PrimaryContent.vue, which keys a componentFor(section) lookup and a parallel propsFor(section) lookup off section.kind, using the SectionKind constant map defined in constants/PrimaryContentSection.js: 

componentFor(section) {
  return {
    ...
    [SectionKind.restEndpoint]: RestEndpoint,
    ...
  }[section.kind];
},
propsFor(section) {
  const { tokens, title, ... } = section;
  return {
    ...
    [SectionKind.restEndpoint]: { tokens, title },
    ...
  }[kind];
},

Unlike restBody, there is no key renaming here — tokens and title pass straight through from the wire section object to component props. The target component is RestEndpoint.vue, whose entire template is two elements: 

<section>
  <LinkableHeading :anchor="anchor">{{ title }}</LinkableHeading>
  <DeclarationSource :tokens="tokens" />
</section>

The heading text is the section's title (in practice always "URL" or "Sandbox URL", see §3), anchored via a local computed property anchor: ({ title }) => anchorize(title). The tokens array is handed directly to DeclarationSource.vue — the same component that renders ordinary Swift/Objective-C declarations sections — which wraps the tokens in a <pre><code> block and maps each one through DeclarationToken.vue. That component's render function switches on token.kind, but its known cases are Swift declaration-token kinds (keyword, identifier, typeIdentifier, attribute, added/removed, etc.) — none of which match the REST-specific kinds method, baseURL, path, or parameter defined for this section's own token schema (see §3). Those four therefore fall through to the default branch of the switch, which renders each token via SyntaxToken.vue: a bare <span class="token-{kind}">{text}</span>. The fifth REST token kind, text, does match a real DeclarationToken case and is instead rendered by RawText.vue as plain text with no wrapping element.

The section also participates in on-this-page navigation: in onThisPageRegistrator.js, restEndpoint is grouped with the other primary-content-anchor kinds (properties, restBody, restCookies, restHeaders, restParameters, restResponses) to register a level-2 on-this-page entry titled from the section's own title field, anchored via anchorize(section.title) — the same anchor computed inside RestEndpoint.vue itself for its LinkableHeading. A dedicated unit test, RestEndpoint.spec.js, confirms the component renders the title through LinkableHeading and forwards the tokens array unmodified to DeclarationSource.

This is a different token-kind vocabulary from the general DeclarationToken.kind enum used by ordinary declarations sections (which has values like keyword/identifier/typeIdentifier and does not include any of these five) — see §1 for the rendering consequence of that mismatch.
EndpointToken.text string yes — The literal text of this token (an HTTP method, a base URL, a path segment, a {parameter}-braced placeholder, or filler text such as the single space between method and URL).

Authoring-side note: there is no Markdown/directive syntax an author writes directly to produce a restEndpoint section. swift-docc's HTTPEndpointSectionTranslator builds the section by reading a symbol's httpEndpointSection (populated from a SymbolGraph symbol mixin, typically produced by converting an OpenAPI specification into a symbol graph upstream of DocC) and tokenizing its method, baseURL (endpoint.baseURL for the production translator instance, or endpoint.sandboxURL for the sandbox instance — the translator is parameterized by an endpointType: RESTEndpointType of .production or .sandbox, which also selects the hard-coded "URL" vs. "Sandbox URL" title), and path, splitting any {curly-brace}-delimited dynamic segments out into their own parameter tokens via a regular expression (tokensFor(method:baseURL:path:), lines 44–77). The underlying plain-data struct, RESTEndpointRenderSection.swift, carries a doc comment with a fully worked example: for the endpoint GET https://www.example.com/api/artists/{id}, the token sequence is (method) GET, (baseURL) https://www.example.com, (path) /api/artists/, (parameter) {id} — matching the fixture in §2 exactly (that fixture additionally has an explicit (text) " " filler token between the method and the base URL, which the doc comment's prose description omits but the translator code inserts at line 49).

DocCArchive transform status: MyHTMLTransformer and MyMarkdownTransformer in Sources/DocCArchive/DocCArchive.swift (lines 242–254) are still stub placeholders (the HTML transformer returns a hardcoded <!-- what? --> comment and the Markdown transformer returns an empty string) — neither has any kind-specific handling for restEndpoint (or any other render section) today. A future implementation would need to render the title as a heading and the token array as a single-line declaration-style code block, mapping each EndpointToken.kind to its own styling (mirroring RestEndpoint.vue and DeclarationToken.vue's default/SyntaxToken fallback path).

4. HTML/CSS mapping in DocC-Render output

Rendered DOM shape, from RestEndpoint.vue's template and the DeclarationSource.vue/DeclarationToken.vue/SyntaxToken.vue chain it delegates to:

  • section — root element (unguarded; always rendered when this section kind is present).
  • Child: whatever root element LinkableHeading renders (an h2 by default, since RestEndpoint.vue does not pass a level prop), with id set to anchorize(title), containing the section's title text (e.g. "URL").
  • Child: pre.source (from DeclarationSource.vue), containing a <code> element (from the shared CodeBlock.vue) which in turn contains one child per token.
  • Each token renders as either a bare text node (kind: "text", via RawText.vue, no wrapping element) or a <span class="token-{kind}">{text}</span> (all four REST-specific kinds — method, baseURL, path, parameter — via SyntaxToken.vue's catch-all render path, since none of them match a case in DeclarationToken.vue's own switch statement).

SCSS: RestEndpoint.vue itself has no <style> block at all — all visual styling comes from the components it delegates to. DeclarationSource.vue's scoped SCSS (importing docc-render/styles/_core.scss) styles the .source root: background via var(--background, var(--color-code-background)), text color via var(--text, var(--color-code-plain)), a bordered/rounded box (border-color: var(--color-grid), border-radius: $large-border-radius), and padding: var(--code-block-style-elements-padding). DeclarationToken.vue's own scoped SCSS defines .token-method { font-weight: $font-weight-bold; } plus rules for token kinds not used here (.token-keyword, .token-string, etc., driven by --syntax-* custom properties) — there is no dedicated .token-baseURL, .token-path, or .token-parameter rule anywhere in that stylesheet, so those three token kinds inherit plain text color from the surrounding .source block.

CSS custom properties consumed: --background/--color-code-background and --text/--color-code-plain (code-block theming), --color-grid (border color), and --code-block-style-elements-padding (interior padding) — all consumed by DeclarationSource.vue's .source rule and shared with every other declaration-style code block in DocC-Render, not specific to restEndpoint itself.

5. Modern CSS-only alternative

NOT APPLICABLE

Current behavior: RestEndpoint.vue is a pure, stateless template — two elements (LinkableHeading, DeclarationSource), both fed directly from props with no v-if, no computed toggle, and no event handler beyond what LinkableHeading itself provides for its permalink-scroll behavior (which is a navigation/anchor concern shared by every heading in DocC-Render, not something introduced by this section kind). The only JavaScript-driven decision specific to this component chain is DeclarationToken.vue's render-time switch on token.kind, which selects which child component to instantiate per token — but that dispatch happens once, at render time, from static data; it is not a runtime state machine that responds to user interaction and has nothing to toggle after the page has rendered.

Because there is no interactive state to hold or reveal/hide in response to a user action, none of the stateful-CSS mechanisms (<details>/<summary>, a checked radio input plus a general sibling selector, :target, the Popover API, <dialog>, :has(), container queries, CSS anchor positioning, view transitions) have anything to attach to here. A static HTML/CSS rendering of this section is simply a heading element followed by a <pre><code> block with per-token <span>s carrying token-{kind} classes — entirely reproducible with plain markup and a handful of static CSS color/weight rules, no scripting or CSS pseudo-state trick required.
