1. How swift-docc-render consumes this

dictionaryExample is one of the block-content kinds dispatched by the shared renderNode() function inside components/ContentNode.vue's BlockType map (dictionaryExample: 'dictionaryExample', line 47). The switch (node.type) case for it is: 

case BlockType.dictionaryExample: {
  const props = {
    example: node.example,
  };
  return createElement(DictionaryExample, { props }, renderChildren(node.summary || []));
}

(src/components/ContentNode.vue#L371-L376). Two things stand out mechanically:

  • Unlike codeListing, image, and video, this case does not inspect node.metadata and never routes through the shared renderFigure helper — a dictionaryExample node is always rendered as a plain DictionaryExample component, even if a metadata.anchor happened to be present on the node (the render node's own metadata field, if any, is simply not read here).
  • node.summary (an optional array of RenderBlockContent) is recursively rendered via renderChildren and passed as the component's default <slot> content, while node.example (the CodeExample object) is passed through untouched as a prop — it is not walked or transformed by ContentNode.vue at all.

The leaf component, components/ContentNode/DictionaryExample.vue, is a thin 36-line wrapper with no local styling of its own: 

<div class="DictionaryExample">
  <slot />
  <CollapsibleCodeListing
    :content="example.content"
    showLineNumbers
  />
</div>

It renders the summary slot content first (typically an introductory paragraph), then delegates the actual code rendering entirely to components/ContentNode/CollapsibleCodeListing.vue, passing example.content (the CodeExample.content array of { collapsible, code } chunks) straight through as that component's content prop, and hard-coding showLineNumbers to true. Note that DictionaryExample.vue does not pass a collapsed prop to CollapsibleCodeListing, so that prop is left at its Vue default (undefined, falsy) — every chunk starts visually expanded regardless of its own collapsible flag; collapsible only adds the .collapsible styling class (a distinct background/text color for those lines), it does not by itself collapse anything in this call site.

CollapsibleCodeListing.vue in turn renders each content chunk as its own <div class="container-general"> inside a single <pre><CodeBlock>…</CodeBlock></pre>, with one <div class="code-line-container"> per source line, each holding an optional <div class="code-number"> gutter (shown because showLineNumbers is always true here) and a <div class="code-line"> with the raw line text (CollapsibleCodeListing.vue#L11-L30). This is a structurally different (and simpler) markup path than the plain codeListing kind's dedicated CodeListing.vue: there is no copy-to-clipboard button, no wrap/lineAnnotations support, and (per the vendored schema in section 3) no per-example syntax-driven client-side syntax highlighting call in this component — highlighting, if any, happens inside the shared CodeBlock child component that wraps the whole listing, not per line.

DocCArchive's own transform layer does not yet do anything with this kind: MyHTMLTransformer and MyMarkdownTransformer (Sources/DocCArchive/DocCArchive.swift:242-254) are still stub placeholders that operate on a whole RenderNode and return a hardcoded HTML comment / empty string respectively, so a dictionaryExample block encountered while walking a parsed archive would decode correctly into Components.Schemas.DictionaryExample but has no dedicated conversion logic in this repository today.

4. HTML/CSS mapping in DocC-Render output

Root element: <div class="DictionaryExample"> (DictionaryExample.vue#L12). It has no scoped <style> block of its own — the file ends after its <script> section, so all visible styling comes from its children:

  • The summary slot content renders as whatever block elements renderChildren produces for those RenderBlockContent nodes (typically a plain <p>).
  • <div class="collapsible-code-listing"> (optionally .single-line when the first chunk has exactly one code line) — the root of CollapsibleCodeListing.vue, styled by its own scoped SCSS (CollapsibleCodeListing.vue#L58-L146).
  • Inside it, <pre><CodeBlock> (the shared components/CodeBlock.vue) wraps one <div class="container-general"> per CodeExampleContent chunk, with modifier classes collapsible (that chunk's collapsible flag) and collapsed (only ever added when both collapsible is true and the component's own collapsed prop is true — never the case for dictionaryExample, see section 1).
  • Each line is a <div class="code-line-container"> containing <div class="code-number"> (a CSS counter-driven line number, always present here since showLineNumbers is hard-coded true) and <div class="code-line"> with the plain line text.

Notable SCSS/CSS from CollapsibleCodeListing.vue's scoped <style lang="scss"> block (imports docc-render/styles/_core.scss): $code-listing-with-numbers-padding and $code-number-padding (spacing variables from src/styles/core/_vars.scss), and a counter-reset: linenumbers / counter-increment: linenumbers pair driving each .code-number::before via CSS content: counter(linenumbers) — the line numbers are generated CSS content, not template-interpolated text. CSS custom properties consumed at paint time: --background / --color-code-background (listing background), --colors-grid / --color-grid (border color), --text / --color-code-plain (default text color), and --color-code-collapsible-background / --color-code-collapsible-text (applied to chunks with collapsible: true).

5. Modern CSS-only alternative

Not applicable

Current behavior, precisely: in the render path actually exercised by dictionaryExample nodes, there is no client-side interactive state at all. DictionaryExample.vue has no data, no event handlers, and no watchers; it statically composes a slot and one child component. CollapsibleCodeListing.vue likewise has no data, methods, or watchers of its own — its only dynamic behavior is the pure-template class binding { collapsible: content.collapsible === true, collapsed: content.collapsible === true && collapsed } on each chunk (line 19-20), and, as established in section 1, the collapsed prop this depends on is never actually passed a truthy value by DictionaryExample.vue, so in practice that class is dead code for this particular kind. The only genuinely dynamic piece anywhere nearby is the line-number gutter, and that is already implemented with a native CSS mechanism today (CSS counters via counter-reset/counter-increment/content: counter(...)), not JavaScript.

There is therefore no interactive behavior here to replace with <details>, :target, the Popover API, :has(), or any other stateful CSS mechanism: this block-content kind renders a static, non-collapsing, non-toggling code listing. (Contrast with the sibling codeListing kind's copy-to-clipboard button and asynchronous syntax highlighter, neither of which is present in this component tree.) If a future revision of DictionaryExample.vue started passing a real collapsed prop through (so that collapsible chunks could be toggled open/closed by a reader), that specific show/hide toggle would be a textbook fit for a checkbox/radio-input + :checked + general-sibling-selector pattern, or a native <details>/<summary> per chunk — but that would be speculating about code that does not exist in the pinned commit, so the verdict for the kind as it actually behaves today is "not applicable."
