1. How swift-docc-render consumes this

intro is a decode-only alias, not a value swift-docc-render's dispatch tables ever see. Both "hero" and "intro" are declared as valid values of RenderSectionKind at RenderSection.swift#L15-L17, and swift-docc's CodableRenderSection decoder accepts either one and routes both to the same IntroRenderSection type: 

switch kind {
case .hero, .intro:
    section = try IntroRenderSection(from: decoder)
...

(CodableRenderSection.swift#L21-L46.) But IntroRenderSection.kind itself is a hardcoded constant, not something set from the decoded value: 

public struct IntroRenderSection: RenderSection, Equatable {
    public let kind: RenderSectionKind = .hero
    ...

(swiftCaseSite's enclosing struct, IntroRenderSection.swift#L12-L13.) So a RenderNode that swift-docc itself produces (via RenderNodeTranslator.visitIntro, RenderNodeTranslator.swift#L294-L295) always re-encodes as "kind": "hero", even if the section originated from JSON that said "kind": "intro". swift-docc's own test suite documents this normalization explicitly, with a comment stating the intent in plain words: "We should always output renderJSON that uses 'hero' to describe an intro section to maintain compatibility" (see the Data sheet section below for the exact test).

Because of that, a grep of swift-docc-render@4a9efad637's src/components/ and src/views/ for the literal string "intro" as a kind discriminator turns up nothing — only unrelated CSS class names (a div.intro wrapper inside Tutorial/Hero.vue, and a .intro class in TutorialsOverview/Chapter.vue) and copy strings like "Watch intro video". The three actual section-kind dispatch tables all key exclusively on hero:

  • Tutorial.vue's SectionComponents map (Tutorial.vue#L46-L66): { assessments: Assessments, hero: Hero, tasks: SectionList, callToAction: CallToAction }.
  • Article.vue's SectionKind map plus componentFor(section) (Article.vue#L45-L124), which maps hero to Article/Hero.vue, a two-line wrapper that re-exports Tutorial/Hero.vue unchanged.
  • TutorialsOverview.vue's inline SectionKind-gated template (<Hero v-if="heroSection" ... />), which renders a separate, unrelated component, TutorialsOverview/Hero.vue.

None of these tables has an intro key, and no v-if/v-else branch anywhere in the searched directories checks section.kind === 'intro'. In other words: even though the wire schema and swift-docc's decoder both still recognize "intro" as an input value (for backward compatibility with older-format render JSON), the current swift-docc-render frontend was not found to contain any code path that would ever receive "kind": "intro" in practice, because swift-docc normalizes it to "hero" before any consumer sees it. This page therefore documents the wire alias and its rendering intent, but the actual template/DOM/SCSS mechanics belong to the sibling page for hero (kinds/section-hero.html), since they are, by construction, the exact same IntroRenderSection payload and the exact same Tutorial/Hero.vue component.

4. HTML/CSS mapping in DocC-Render output

There is no CSS or DOM structure specific to the "intro" spelling — because swift-docc normalizes "intro" to "hero" before the render JSON is written to data/ in a .doccarchive, swift-docc-render's dispatch tables (Section 1) never branch on kind === "intro", so there is no distinct template, class, or SCSS rule to document that differs from the hero case. The full DOM shape (root LinkableSection anchor id="introduction", div.hero.dark, div.bg, H1, div.intro, div.metadata) and the SCSS custom properties it consumes (--color-tutorial-hero-background, --color-tutorial-hero-text, --color-hero-eyebrow, from src/styles/core/colors/_light.scss) are documented on the hero page, Section 4 and apply identically here, since they are produced by the same Tutorial/Hero.vue component regardless of which spelling appeared in the source JSON before swift-docc's decode step.

DocCArchive transform status: MyHTMLTransformer and MyMarkdownTransformer in Sources/DocCArchive/DocCArchive.swift are still stub placeholders (the HTML transformer returns a hardcoded <!-- what? --> comment and the Markdown transformer returns an empty string) — there is no kind-specific handling for intro/hero/ IntroRenderSection today, and because DocCArchive decodes with the generated type (which only exposes title/kind/content), it also could not currently tell an incoming "intro"-spelled section apart from any other metadata beyond that.

5. Modern CSS-only alternative

NOT APPLICABLE

Current behavior in play: "intro" is not a runtime UI state at all — it is a JSON string discriminator that exists only inside swift-docc's Swift decoder (CodableRenderSection.init(from:)'s switch kind { case .hero, .intro: ... }), and it is fully resolved (normalized to "hero") before any HTML, Vue component, or CSS is ever produced. There is no client-side branching, toggle, visibility state, or DOM structure that depends on whether the original JSON said "hero" or "intro"; by the time swift-docc-render's ContentNode/Tutorial.vue/Article.vue code runs, only "hero" is possible. Since there is no rendering-time behavior distinguishing these two spellings to replace, no native CSS mechanism (<details>, radio+:checked, :target, Popover API, <dialog>, :has(), container queries, anchor positioning, view-transitions) is relevant to this distinction. (The genuinely stateful piece of the rendered output — the "Watch intro video" modal inside Tutorial/Hero.vue — is identical regardless of the intro/hero spelling and is analyzed on the hero page, Section 5, which reaches a partial verdict for that unrelated interactive behavior via <dialog>.)
