1. How swift-docc-render consumes this

Short answer, stated honestly up front: a real, exhaustive grep of swift-docc-render's src/components/ and src/views/ trees (including src/components/Article/, src/components/Tutorial/, src/components/TutorialsOverview/, and src/views/) for the literal string "contentAndMediaGroup" and for the case-insensitive substring contentandmediagroup found zero matches. No Vue component in this renderer dispatches on, imports, or otherwise names this kind. This is not a gap in the search — it reflects the actual state of the swift-docc translator, traced below.

contentAndMediaGroup is declared as a case of RenderSectionKind and backed by a real Swift payload type, ContentAndMediaGroupSection, which wraps an array of the singular ContentAndMediaSection plus a shared layout. It is registered in the decode dispatch table (CodableRenderSection.swift) and in the JSON-diffing machinery (AnyRenderSection.swift), so swift-docc can decode and diff it if a document happens to contain it.

But grepping the full RenderNodeTranslator.swift for ContentAndMediaGroupSection( — the constructor call that would have to run for this kind to ever be emitted — turns up nothing. The translator's actual tutorial-overview path is:

(RenderNodeTranslator.swift#L504-L536) VolumeRenderSection.Chapter.content is a flat [RenderBlockContent], not a ContentAndMediaGroupSection. Likewise, the singular visitContentAndMedia(_:) — the function that actually constructs ContentAndMediaSection values — is called directly for tutorial-page @Section/@Stack content (RenderNodeTranslator.swift#L1006-L1014), and its results are stored one at a time as TutorialSectionsRenderSection.Section.contentSection (RenderNodeTranslator.swift#L155) — never batched into a contentAndMediaGroup wrapper. In short: contentAndMediaGroup is a decodable-but-dead kind in the current pipeline — present in the enum and in decode/diff plumbing for backward compatibility, but not constructed by any code path that produces new render trees, and consequently never something a current DocC build writes into a .doccarchive, and never something swift-docc-render needs to draw.

Because there is no render-side consumer, this page cannot cite a component, template, or SCSS file for contentAndMediaGroup itself in section 4 below; it instead documents where the closely related, actually-rendered singular contentAndMedia kind is consumed, for context (see the separate contentAndMedia reference page for the full Vue-side treatment).

4. HTML/CSS mapping in DocC-Render output

There is no HTML/CSS mapping to document for contentAndMediaGroup itself: no template, class, id, or SCSS partial in swift-docc-render names or dispatches on this kind (see section 1). There is nothing to map because there is nothing rendered.

For descriptive context only: the individual contentAndMedia sections that this kind would group are, when they occur directly (unwrapped) in tutorial content, rendered by ContentAndMedia.vue as div.content-and-media (with modifier classes media-leading / media-trailing), using SCSS from that component's own <style scoped lang="scss"> block and the shared docc-render/styles/_core.scss partial. See the dedicated contentAndMedia reference page for the full breakdown of that component; none of it is specific to the group wrapper, which simply does not exist on the render side.
