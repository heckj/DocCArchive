1. How swift-docc-render consumes this

This is one of the "primary content" section kinds attached to a symbol's primaryContentSections array — specifically the one used for documenting a property-list key or entitlement key (for example, an Info.plist key or a .entitlements key). The dispatch happens in PrimaryContent.vue, which keys a lookup table on section.kind via the SectionKind constant map (defined in constants/PrimaryContentSection.js, where details: 'details' is the literal wire value). For SectionKind.details it selects the PropertyListKeyDetails component and passes through the section's details object as a prop: 

componentFor(section) {
  return {
    ...
    [SectionKind.details]: PropertyListKeyDetails,
    ...
  }[section.kind];
},
propsFor(section) {
  const { ..., details, ... } = section;
  return {
    ...
    [SectionKind.details]: { details },
    ...
  }[kind];
},

The actual rendering happens in PropertyListKeyDetails.vue. Its template root is a <section class="details"> containing a heading (via the shared LinkableHeading component, anchored to "details" per ContentSectionAnchors.js) followed by a single definition list (<dl>) that conditionally emits up to three key/value pairs depending on the details.titleStyle field: 

<section class="details">
  <LinkableHeading :anchor="contentSectionData.anchor">
    {{ $t(contentSectionData.title) }}
  </LinkableHeading>
  <dl>
    <template v-if="isSymbol">
      <dt class="detail-type">{{ $t('metadata.details.name') }}</dt>
      <dd class="detail-content">{{ details.ideTitle }}</dd>
    </template>
    <template v-if="isTitle">
      <dt class="detail-type">{{ $t('metadata.details.key') }}</dt>
      <dd class="detail-content">{{ details.name }}</dd>
    </template>
    <dt class="detail-type">{{ $t('metadata.details.type') }}</dt>
    <dd class="detail-content">
      <PropertyListKeyType :types="details.value" />
    </dd>
  </dl>
</section>

The isSymbol/isTitle computed properties read details.titleStyle ("symbol" vs. "title") together with the presence of details.ideTitle to decide which extra row to show: when the raw key itself is the displayed title (titleStyle === 'symbol'), it shows a "Name" row with the human-friendly ideTitle; when a friendly display name is the title instead (titleStyle === 'title'), it shows a "Key" row with the raw details.name. The "Type" row always renders, delegating to a small child component, PropertyListKeyType.vue, which turns the value array of { baseType, arrayMode } objects into an English phrase such as "array of strings or a number" (joining with commas and a trailing "or", and pluralizing baseType when arrayMode is true).

See also: Localization fallback — how $t()/locale-key lookup works across every kind that uses it.

Source-of-truth alignment: swift-docc's own Swift model for this section lives at Sources/SwiftDocC/Model/Rendering/Symbol/PropertyListDetailsRenderSection.swift (lines 27–70), defining public struct PropertyListDetailsRenderSection: RenderSection, Equatable with a nested Details struct. The struct's field names (rawKey, value, platforms, displayName, titleStyle) differ cosmetically from the vendored JSON Schema/generated Swift names (name, value, platforms, ideTitle, titleStyle) purely via CodingKeys remapping — the wire JSON is identical either way. 4. HTML/CSS mapping in DocC-Render output

  • <section class="details"> — root element, scoped styles live entirely inline in PropertyListKeyDetails.vue's own <style scoped lang="scss"> block (there is no separate dedicated SCSS partial for this kind).
  • Heading — rendered by the shared LinkableHeading component (src/components/ContentNode/LinkableHeading.vue) as an <h2> (per ContentSectionAnchors.js's level: 2 for SectionKind.details), with id="details" set from the anchor prop, enabling deep-linking such as #details.
  • <dl> — the key/value list; no dedicated class.
  • <dt class="detail-type"> — the localized row label ("Name", "Key", or "Type"); styled with font-weight: $font-weight-semibold, padding-left: 1rem, and padding-top: var(--spacing-param) (collapsed to 0 for the first item via &:first-child).
  • <dd class="detail-content"> — the row's value (plain text, or the nested <div class="type"> from PropertyListKeyType.vue for the Type row); styled with padding-left: 2rem.
  • Responsive behavior: at the small breakpoint (mixin @include breakpoint(small) from the shared docc-render/styles/_core.scss import), the left padding on both .detail-type and .detail-content collapses to 0.
  • PropertyListKeyType.vue contributes its own tiny scoped rule: .type::first-letter { text-transform: capitalize; }, capitalizing the first letter of the generated English type phrase.
  • CSS custom property consumed: --spacing-param, declared globally in src/styles/base/_typography.scss (--spacing-param: #{rem(28px)}) and overridden per-view in DocumentationTopic.vue's root style block to track the surrounding topic page's stacked-content spacing scale.
  • SCSS files involved: PropertyListKeyDetails.vue's own scoped block, PropertyListKeyType.vue's own scoped block, and the shared docc-render/styles/_core.scss import (for the breakpoint mixin and font-weight variable).

5. Modern CSS-only alternative

Not applicable

Current behavior: PropertyListKeyDetails.vue holds no component-local reactive state and has no click/toggle/expand-collapse handlers. It is a one-way template render: two computed properties (isSymbol, isTitle) derive a boolean directly from the immutable details.titleStyle/details.ideTitle props passed down from the parent, and v-if uses those booleans to decide, at render time, which of two mutually exclusive rows to include. This is conceptually a compile-time/data-driven conditional, not a runtime UI toggle a user can flip — there is no user interaction anywhere in this component (no click handler, no v-model, no v-show tied to a click). PropertyListKeyType.vue is likewise a pure, stateless string-formatting computed chain with no interactivity.

Because there is no interactive state to reproduce (no expand/collapse, no active/inactive selection, no show/hide driven by user input), none of the native state-holding mechanisms (<details>/<summary>, radio+:checked, :target, the Popover API, <dialog>, :has(), container queries, CSS anchor positioning, view-transitions) have anything to hold or replace for this kind. The one piece of "state" on the page — which of the two conditional rows is shown — is fully determined by the server-rendered JSON payload before the page ever reaches the browser, so it is not something CSS (native or otherwise) would ever need to manage.
