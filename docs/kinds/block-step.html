1. How swift-docc-render consumes this

Unlike most block-tier kinds, step is not handled inside the generic renderNode() dispatch table in ContentNode.vue — a grep of that file for "step"/BlockType.step turns up nothing. Instead, step nodes only ever appear inside the content array of a tutorial section, and a dedicated component, SectionSteps.vue, walks that array itself and special-cases the step type directly, via its own isStepNode helper at SectionSteps.vue#L164-L166: 

isStepNode({ type }) {
  return type === 'step';
},

SectionSteps is registered as the Steps component in Section.vue#L28-L39 and mounted whenever a tutorial section's stepsSection content is non-empty (Section.vue#L18-L24). Its contentNodes computed property (SectionSteps.vue#L122-L149) reduces over the raw content array: any node where isStepNode is true is assigned the Step component (with a running stepNumber counter, its array index, numberOfSteps, and sectionNumber injected as props); every other node (interstitial prose, asides, images between steps) falls through to a plain ContentNode for generic block rendering. So the dispatch on the literal wire value "step" happens once, in this reduce, rather than in a switch statement.

Each matched node is rendered by Step.vue. Its template root is a <div class="step-container"> wrapping a <div class="step"> that holds a localized "Step {number}" label followed by a nested <ContentNode :content="content"> for the step's instructional body, and (conditionally) a second <ContentNode> for the caption (Step.vue#L11-L22). Alongside that, a sibling <div class="media-container"> (Step.vue#L25-L42) renders the step's media reference as an <Asset> and its code reference (paired with an optional runtimePreview) as a <MobileCodePreview> — but that container is CSS-hidden except at the small breakpoint, because on wider viewports the same media/code/runtimePreview triple is instead shown once, on the side, by the parent.

That "on the side" rendering is SectionSteps.vue's own job: it keeps a single visibleAsset ({ media, code, runtimePreview }) in component data, seeded from the first step (SectionSteps.vue#L93-L111), and swaps it via onFocus(key) (SectionSteps.vue#L186-L199) whenever the "active" step changes. The active step index (activeStep, passed down to every Step as currentIndex) is driven by scroll position: SectionSteps mixes in onIntersect and registers each step's DOM node as an IntersectionObserver target (getIntersectionTargets, SectionSteps.vue#L233-L235), with an observer rootMargin of -35% 0% -65% 0% (SectionSteps.vue#L76) that treats roughly the middle third of the viewport as the "focus band"; whichever step's container element crosses into that band calls onIntersect (SectionSteps.vue#L241-L255), which updates activeStep. Inside Step.vue, that prop feeds a computed isActive: ({ index, currentIndex }) => index === currentIndex (Step.vue#L116), which toggles the .focused class on the step's root <div class="step"> (Step.vue#L13-L18), producing the highlighted left-border treatment for whichever step the reader has scrolled to.

An author produces this by writing a @Step directive inside a @Steps group:

@Steps {
  @Step {
    Hello, world!

    > Step caption

    @Image(source: media.png, alt: "...")
    @Code(name: "Code", file: main.swift)
  }
}

Per swift-docc's directive parser (Sources/SwiftDocC/Semantics/Tutorial/Tasks/Steps/Step.swift:89-105), the first paragraph becomes content, an immediately following second paragraph (typically a block quote in practice) becomes caption, and at most one media directive plus one @Code directive supply media/code; runtimePreview is populated later by the compiler when it can produce a live preview of the referenced code file, not by anything the author writes directly.

4. HTML/CSS mapping in DocC-Render output

Per step, the DOM shape (from Step.vue's template) is:

  • div.step-container plus a per-index modifier class step-{stepNumber} — the outer wrapper, controls vertical spacing between consecutive steps.
  • div.step (with data-index and, conditionally, class .focused) — the visible card; carries the rounded background, padding, and the left accent border used for the active-step highlight.
  • p.step-label — the localized "Step N" text.
  • a nested ContentNode render tree for content (no extra wrapper class beyond whatever ContentNode itself emits), and, when non-empty, a second ContentNode tree wrapped in .caption for the step's caption.
  • div.media-container (hidden above the small breakpoint) holding an Asset for media and a MobileCodePreview for code/runtimePreview.

The wider, non-mobile media/code panel lives outside Step.vue entirely, in SectionSteps.vue's template: div.steps > div.asset-container.asset-wrapper (via BackgroundTheme), holding either an Asset (.step-asset) or a CodePreview, whichever the currently-focused step's visibleAsset supplies.

Styling is split across two scoped SCSS blocks, both lang="scss" in the same files:

  • Step.vue's <style scoped> (lines 121–222) — defines .step-container, .step, .step-label, .caption, and the small-breakpoint .media-container rules, including the ::before pseudo-element used for the focused-state accent bar.
  • SectionSteps.vue's <style scoped> (lines 274–515) — defines .steps, .content-container, .asset-container, .asset-wrapper, .interstitial, and the .fade-* transition classes used when visibleAsset swaps.

CSS custom properties consumed (from src/styles/core/colors/_light.scss and friends):

  • --color-step-background (aliases --color-fill-secondary) — the step card's background, and the "ide" asset-wrapper background.
  • --color-step-focused (aliases --color-figure-light-gray) — both the border and background color of the .step::before accent bar shown when .focused is applied.
  • --color-step-text (aliases --color-figure-gray-secondary) — fallback color for .step-label, overridden by --colors-text when set.
  • --color-step-caption (aliases --color-figure-gray-secondary) — the caption's top border and text color.

Non-custom-property SCSS variables also feed in: $step-padding-horizontal (2rem, defined in src/styles/core/_vars.scss) sets the card's horizontal padding, and $step-border-width sets the width of the ::before accent bar.

5. Modern CSS-only alternative

Partial

Current behavior: the "active step" highlight is not a click-driven toggle but a scroll-position computation. SectionSteps.vue registers each step element with an IntersectionObserver whose rootMargin is -35% 0% -65% 0% (SectionSteps.vue#L76), so only elements crossing the vertical middle third of the viewport fire; the observer callback (onIntersect) writes a single reactive activeStep index in component data, which is handed down as the currentIndex prop to every Step, where isActive: index === currentIndex toggles the .focused class (Step.vue#L116). The same onFocus call also swaps visibleAsset, which is what actually keeps the media/code panel in sync with the highlighted step.

This is a case where a purely declarative CSS mechanism can replace part of the behavior but not all of it. CSS Scroll-driven Animations' view() timeline (an animation-timeline: view() tied to each .step element, animating e.g. its border-color/opacity as the element crosses a declared view-timeline-inset band) can mechanically reproduce the .focused visual toggle without any IntersectionObserver or Vue reactive state at all — the browser itself tracks each element's position in the scrollport and drives the animation, which is exactly the "highlight whichever card is in the middle third" effect the JS produces today. That covers the .step card's own highlight.

What it cannot cover: the side-panel synchronization. The visibleAsset swap is cross-element state — the currently-scrolled-to step's media/code/runtimePreview must be reflected in a different DOM subtree (the sticky .asset-container that lives outside any individual step). No scroll-linked CSS mechanism can select "the sibling asset panel corresponding to whichever card is currently in view" and swap its content purely in CSS: :has() only looks at descendants/siblings within a single selector chain relative to a fixed anchor, not "whichever of N possible panels matches the one whose trigger is currently intersecting," and there is no CSS construct that reads a scroll-timeline position in one element and uses it to pick which of several unrelated elements to display. That part of the behavior remains inherently JavaScript-driven (or would require, at minimum, per-step CSS anchor-positioning + :target-style plumbing that swift-docc-render's markup does not currently provide hooks for), which is why this is scored "Partial" rather than "Applicable."
