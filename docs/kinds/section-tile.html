1. How swift-docc-render consumes this

Important caveat found during research: tile is declared as a case of RenderSectionKind in swift-docc (Sources/SwiftDocC/Model/Rendering/RenderSection.swift:17, case hero, intro, tasks, assessments, volume, contentAndMedia, contentAndMediaGroup, callToAction, tile, articleBody, resources), but there is no RenderSection-conforming struct anywhere in swift-docc whose kind is "tile", and no "kind": "tile" literal appears anywhere in the wire format. Searching AnyRenderSection.swift and CodableRenderSection.swift (the two places that switch over every RenderSectionKind case to decode a section) turns up no reference to .tile at all. The enum case appears to be reserved/vestigial.

What DocC actually emits under the name "tile" is RenderTile (Sources/SwiftDocC/Model/Rendering/Tutorials Overview/Resources/RenderTile.swift) — a plain Codable struct with no kind field of its own. It is never a top-level section; it only appears as an element of the tiles array inside ResourcesRenderSection, whose own discriminator is "kind": "resources", not "tile". This page therefore documents RenderTile/tile as an embedded object type, consistent with how the JSON schema (step 2 below) models it — as an inline, untagged object under ResourcesRenderSection.tiles, not as its own discriminated section.

Tracing how swift-docc-render actually renders this data, starting from the top of a tutorial overview page: LearningPath.vue receives the render node's top-level sections array and keeps a small SectionKind lookup ({ resources: 'resources', volume: 'volume' }). Anything whose section.kind is "resources" is routed to Resources.vue, and only content and tiles are forwarded as props: 

const SectionKind = { resources: 'resources', volume: 'volume' };
...
componentFor: ({ kind }) => ({
  [SectionKind.resources]: Resources,
  [SectionKind.volume]: Volume,
}[kind]),
propsFor: ({ chapters, content, image, kind, name, tiles }) => ({
  [SectionKind.resources]: { content, tiles },
  [SectionKind.volume]: { chapters, content, image, name },
}[kind]),

See LearningPath.vue#L38-L105 for the dispatch table and prop-forwarding.

Resources.vue's template is a two-line wrapper: a VolumeName heading (labelled "Resources") followed by a TileGroup bound directly to the tiles prop: 

<section class="resources" id="resources" tabindex="-1">
  <VolumeName :name="$t('sections.resources')" :content="content" />
  <TileGroup :tiles="tiles" />
</section>

See Resources.vue#L11-L59. TileGroup is a local alias for ResourcesTileGroup.vue.

ResourcesTileGroup.vue loops the array with v-for="tile in tiles", forwarding exactly the four fields the wire format defines (action, content, identifier, title) as props to one Tile component instance per array element — this is the actual per-item dispatch site for an individual tile object: 

<div class="tile-group" :class="countClass">
  <Tile v-for="tile in tiles" v-bind="propsFor(tile)" :key="tile.title" />
</div>
...
propsFor: ({ action, content, identifier, title }) => ({ action, content, identifier, title }),

See ResourcesTileGroup.vue#L11-L46. Tile here is a local alias for ResourcesTile.vue — the component that actually renders one tile's markup.

ResourcesTile.vue renders the tile's icon (chosen from identifier), title, body content (delegated to the generic ContentNode block-content renderer), and an optional call-to-action link: 

<div class="tile">
  <div v-if="identifier" class="icon">
    <component :is="iconComponent" />
  </div>
  <div class="title">{{ title }}</div>
  <ContentNode :content="content" />
  <DestinationDataProvider v-if="action" :destination="action" v-slot="{ url, title }">
    <Reference class="link" :url="url">
      {{ title }}
      <InlineChevronRightIcon class="link-icon icon-inline" />
    </Reference>
  </DestinationDataProvider>
</div>

See ResourcesTile.vue#L11-L96. identifier selects the icon component from a fixed map (documentation→DocumentIcon, downloads→DownloadIcon, forums→ForumIcon, sampleCode→CurlyBracketsIcon, videos→PlayIcon; the sixth wire value, featured, has no icon and falls through to undefined, so no icon renders). action, when present, is resolved through DestinationDataProvider/Reference to a real link with a trailing chevron icon.

Note: there is a separate, unrelated .tile CSS class in Volume.vue (a "chapter" card in the same tutorials-overview page) — same class name, different component and different wire kind (volume.chapters[], not ResourcesRenderSection.tiles[]). It is not connected to this page's tile and is out of scope here.

See also: Localization fallback — how $t()/locale-key lookup works across every kind that uses it.

3. Data sheet

There is no standalone TileRenderSection schema object. The relevant schema is ResourcesRenderSection in Sources/DocCArchive/Vendored/RenderNode.spec.json lines 360-406, whose tiles field is an array of an inline, unnamed object schema (no $ref) with no kind/discriminator property of its own. The corresponding generated Swift type in Sources/DocCArchive/generated/Types.swift lines 3466-3546 is Components.Schemas.ResourcesRenderSection, and the per-tile type — because the schema gave it no name — is synthesized by swift-openapi-generator as the nested type Components.Schemas.ResourcesRenderSection.TilesPayloadPayload.

ResourcesRenderSection (the enclosing section)

FieldTypeRequiredEnum / defaultNote
kindstringyes"resources" (only value)Section discriminator; generated as KindPayload single-case enum. Not "tile".
tilesarray of inline objectyes—The array documented below; generated as TilesPayload = [TilesPayloadPayload].

ResourcesRenderSection.tiles[] (the "tile" object — TilesPayloadPayload)

FieldTypeRequiredEnum / defaultNote
identifierstringyes"featured" | "documentation" | "sampleCode" | "downloads" | "videos" | "forums"Generated as IdentifierPayload; selects the icon component in ResourcesTile.vue (no icon for featured).
titlestringyes—Rendered verbatim as the tile heading.
contentarray of RenderBlockContentyes—Tile body, delegated to ContentNode.
actionRenderInlineContentno—Call-to-action link/reference; when present, rendered as a chevron-suffixed link via DestinationDataProvider.
mediastring (reference id), nullableno—Declared in the JSON schema ("format": "reference", "nullable": true); generated as an optional Swift.String?. Not read anywhere in ResourcesTile.vue/ResourcesTileGroup.vue/Resources.vue — present in the wire format but currently unused by the render UI.

Source: Sources/DocCArchive/Vendored/RenderNode.spec.json lines 360-406 (ResourcesRenderSection, including the inline tiles item schema); generated types at Sources/DocCArchive/generated/Types.swift lines 3466-3546. For comparison, the authoring-side Swift model in swift-docc is RenderTile (Sources/SwiftDocC/Model/Rendering/Tutorials Overview/Resources/RenderTile.swift), which has the identical four fields (identifier, title, content, action, media) plus its own nested Identifier enum with the same six raw values.

4. HTML/CSS mapping in DocC-Render output

  • section#resources.resources — root of the whole resources block, rendered by Resources.vue; carries a scroll-margin tabindex="-1" for the on-page navigation anchor.
  • .tile-group (on the wrapping <div> from ResourcesTileGroup.vue) — a CSS grid whose column/row layout is chosen entirely by a count-N modifier class (count-1 through count-6, one class per possible tile count 1-6), each with its own grid-template-columns/grid-template-rows; e.g. count-5 uses a 6-column grid where the first two tiles span 3 columns and the rest span 2. $tile-margin comes from the SCSS variable $tutorials-overview-tile-margin-single-side (src/styles/core/_vars.scss:29, currently 2px).
  • .tile (on each tile's root <div>, from ResourcesTile.vue) — background var(--color-tutorials-overview-fill-secondary, dark-color(fill-secondary)), text color var(--color-tutorials-overview-content-alt), fixed padding.
  • .icon — fixed 25px square icon slot; SVG fill uses var(--color-tutorials-overview-icon), and any stroke paths use var(--color-tutorials-overview-content-alt).
  • .title — tile heading, font-styles(tutorials-overview-chapter-title).
  • .content — tile body wrapper around the nested ContentNode output; un-ordered lists inside it have bullets suppressed (list-style-type: none) with a zero-width space injected via li:before purely to keep list semantics for assistive tech.
  • .link / .link-icon — the optional call-to-action row, colored via var(--colors-link, var(--color-tutorials-overview-link)), with a small trailing InlineChevronRightIcon.
  • Breakpoint behavior: at the small breakpoint every count-N grid collapses to a single column (grid-template-columns: 1fr); between the "medium" and "nav-medium" breakpoints the grid is similarly forced to one column.
  • SCSS files involved: ResourcesTileGroup.vue (scoped style block, the grid/count-N rules), ResourcesTile.vue (scoped style block, the card look), plus shared src/styles/core/_vars.scss (tile margin) and src/styles/core/colors/_light.scss / _dark.scss (the color custom properties below).
  • CSS custom properties consumed: --color-tutorials-overview-fill-secondary, --color-tutorials-overview-content-alt, --color-tutorials-overview-icon, --colors-link/--color-tutorials-overview-link.

5. Modern CSS-only alternative

Not applicable

Current behavior in play: none of the three components involved (ResourcesTileGroup.vue, ResourcesTile.vue, Resources.vue) hold any component data(), watch any reactive state, or attach any event listeners. ResourcesTileGroup.vue's only logic is a computed countClass derived once from the static prop tiles.length, and ResourcesTile.vue's only logic is a computed iconComponent lookup from the static prop identifier. Both are pure, one-shot derivations from server-supplied JSON at mount time — there is no click handler, toggle, tab switch, or any other run-time interaction to hold state for.

Because there is no interactive behavior in this kind at all — it is a static, read-only grid of link cards — there is no state-holding mechanism (native or JS) to replace. The grid layout itself (grid-template-columns chosen by a count-N class) is already plain CSS, not a candidate for further "modernization" via :has(), <details>, or any other native interactive primitive; those mechanisms exist to replace scripted state transitions, and this kind has none.
