Not Found (404)

Route: fallback * → views/NotFound.vue

1. Route & dispatch · 2. Chrome · 3. Hierarchy tree
This page has no RenderNode-schema content at all — there is no data/*.json file behind it. NotFound.vue renders a single static, localized string ($t('error.not-found'), English value "The page you're looking for can't be found.") as an <h1> inside the shared GenericError component. No other props, no illustration, no links back into the site are rendered by the framework itself.

1. Route and component dispatch

src/routes.js declares two fallbackRoutes, both using Vue Router's catch-all path: '*'. The first entry is this one: 

{
  path: '*',
  name: notFoundRouteName,
  component: NotFound,
}

notFoundRouteName is the string constant 'not-found', defined in src/constants/router.js. Because this route is registered after the real page routes (tutorials-overview, topic, documentation-topic) in the combined route table ([...pagesRoutes, ...fallbackRoutes]), Vue Router only matches it when no other route pattern matches the current URL — i.e. this is a genuine, ordinarily reachable 404: any URL that doesn't start with /tutorials/... or /documentation... lands here.

The matched component is views/NotFound.vue. Its template is just: 

<GenericError :message="$t('error.not-found')">
  <slot />
</GenericError>

The English locale string (src/lang/locales/en-US.json, under error) is "not-found": "The page you're looking for can't be found." — this is the entire user-visible message. The component also has a created() hook that calls AppStore.setAllLocalesAreAvailable() and route guards (beforeRouteEnter/beforeRouteUpdate) that call updateLocale; these affect the language switcher's available-locales state, not anything rendered on this page itself. NotFound accepts a default <slot /> but nothing in the codebase passes content into it, so in practice it is always empty.

2. Page chrome

There is no chrome on this page: no top navigation bar, no sidebar, no search field, no breadcrumb. The entire page body is whatever components/GenericError.vue renders: 

<div class="generic-error">
  <div class="container">
    <h1 class="title error-content">{{message || $t('error.unknown')}}</h1>
    <slot />
  </div>
</div>

For this page, message is always supplied ($t('error.not-found')), so the $t('error.unknown') fallback ("An unknown error occurred.") is never reached here — it only matters for ServerError.vue, which does not pass a message prop at all. There is no icon, no illustration, and no "go home" link; the <slot /> passed through from NotFound.vue is empty, so nothing renders below the heading.

CSS (scoped in GenericError.vue, importing docc-render/styles/_core.scss): .container applies the shared section-content mixin (centers and caps the content width) plus margin-top: 92px / margin-bottom: 140px to vertically place the message roughly mid-page below where a header would otherwise sit. .error-content fixes the heading to a 502px width (narrowing at the medium/small breakpoints to 420px/330px) and centers it with margin-left/right: auto. .title centers the text (text-align: center) and applies the shared headline font-style mixin. Net effect: a single centered heading, vertically offset from the top of an otherwise blank page, with no other visual elements.
