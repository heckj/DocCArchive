Route: fallback * (unreachable by normal navigation) → views/ServerError.vue

1. Route & dispatch · 2. Chrome · 3. Hierarchy tree
This page has no RenderNode-schema content at all — there is no data/*.json file behind it, and unlike the 404 page, no real URL ever routes here during ordinary browsing. ServerError.vue renders a single static, localized string ($t('error.unknown'), English value "An unknown error occurred.") as an <h1> inside the same shared GenericError component used by NotFound.vue. No other props, no illustration, no links back into the site are rendered by the framework itself.

1. Route and component dispatch

src/routes.js declares this as the second of two fallbackRoutes, both registered with Vue Router's catch-all path: '*': 

{
  path: '*', // purposefully unreachable without a forced navigation
  name: serverErrorRouteName,
  component: ServerError,
}

The source comment is explicit: this route is purposefully unreachable without a forced navigation. Because Vue Router resolves catch-all routes in registration order and this entry is listed second (after the not-found catch-all, which is itself listed after the real page routes), the not-found route's identical path: '*' pattern always wins the match first for any ordinary unmatched URL. serverErrorRouteName (the string constant 'server-error', from src/constants/router.js) can therefore only be reached by code that explicitly navigates to it by name (e.g. router.push({ name: serverErrorRouteName }) from some runtime error-handling path), not by any URL a user could type or link to.

The matched component is views/ServerError.vue. Its template is just: 

<GenericError />

Note the difference from NotFound.vue: no message prop is passed here, so GenericError falls back to its own default, $t('error.unknown') — English value "An unknown error occurred." (src/lang/locales/en-US.json, under error). The component otherwise has the identical shape as NotFound.vue: a created() hook calling AppStore.setAllLocalesAreAvailable() and the same beforeRouteEnter/beforeRouteUpdate locale-update guards, none of which affect what's rendered on the page itself. Unlike NotFound.vue, ServerError.vue does not declare a default <slot /> in its own template at all — it renders <GenericError /> with no children, so GenericError's internal <slot /> is empty here too, just via a different mechanism (no slot content ever offered, rather than an empty slot being passed through).

2. Page chrome

There is no chrome on this page: no top navigation bar, no sidebar, no search field, no breadcrumb. The entire page body is whatever components/GenericError.vue renders — the exact same component NotFound.vue uses: 

<div class="generic-error">
  <div class="container">
    <h1 class="title error-content">{{message || $t('error.unknown')}}</h1>
    <slot />
  </div>
</div>

Here message is undefined (ServerError.vue passes no prop), so the template's message || $t('error.unknown') expression falls through to the default string. There is no icon, no illustration, and no "go home" link; the inner <slot /> receives nothing, so nothing renders below the heading.

CSS is identical to the 404 page, since it is the same scoped stylesheet on the same shared component: .container applies the shared section-content mixin plus margin-top: 92px / margin-bottom: 140px; .error-content caps the heading to 502px wide (narrowing at medium/small breakpoints), centered via margin-left/right: auto; .title centers the text and applies the shared headline font-style mixin. Net effect: a single centered heading, vertically offset from the top of an otherwise blank page.
