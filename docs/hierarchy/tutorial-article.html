1. Route and component dispatch

Any path under /tutorials is served by views/Topic.vue, a thin dispatcher that renders one dynamic <component> based on the fetched node's top-level kind. It defines a two-entry lookup table: 

const TopicKind = {
  article: 'article',
  tutorial: 'project',
};
...
componentFor(topic) {
  const { kind } = topic;
  return {
    [TopicKind.article]: Article,
    [TopicKind.tutorial]: Tutorial,
  }[kind];
},

(Topic.vue#L37-L40 and Topic.vue#L109-L115.) A wire kind: "article" node — this page's subject, a free-standing tutorial-catalog article, distinct from a numbered @Tutorial project (wire kind: "project", which dispatches to Tutorial instead) — resolves to components/Article.vue. This is a completely different rendering path from the DocumentationTopic.vue that handles a DocumentationRenderNode's own, same-string kind: "article" value — see the note on kinds/node-article.html, which already covers this ambiguity in detail.

Topic.vue's propsFor(topic) forwards the same flattened prop set to either branch — hierarchy, metadata, references, sections, and identifierUrl (from identifier.url) — so Article.vue receives exactly: 

props: {
  hierarchy: { type: Object, required: true },
  metadata: { type: Object, required: true },
  references: { type: Object, required: true },
  sections: {
    type: Array,
    required: true,
    validator: sections => sections.every(({ kind }) => (
      Object.prototype.hasOwnProperty.call(SectionKind, kind)
    )),
  },
  identifierUrl: { type: String, required: true },
}

(Article.vue#L69-L93.) The wire schema backing this is TutorialsPageRenderNode with kind == .article — not DocumentationRenderNode — so an article page's sections array (not primaryContentSections) is what actually drives the page body, per section 4 below.

2. Page chrome (non-schema)

Article.vue renders exactly one piece of chrome above its section list: NavigationBar, imported as theme/components/Tutorial/NavigationBar.vue — the identical component path (and identical props: chapters, technology, topic, rootReference, identifierUrl) used by the Tutorial (Project) page. There is no separate "article navigation bar" component; the same dropdown-based nav chrome (technology title, primary chapter dropdown, secondary in-page-section dropdown, mobile dropdown fallback) appears on both page types, gated only by v-if="!isTargetIDE" (Article.vue#L13-L20).

As on the Tutorial (Project) page, there is no sidebar and no search chrome on this page type — both are exclusively part of the Documentation Topic page's DocumentationLayout shell, which an article/project-kind tutorial page never mounts. The only other non-schema DOM is a <main id="app-main"> wrapper and a PortalTarget name="modal-destination" used by nested components (e.g. the hero's "Watch intro video" modal) to portal modal markup to (Article.vue#L21-L30).

          • BLOCK p — paragraph

        • CHROME HeroMetadata — project-files link / estimated time / Xcode requirement chips

        • CHROME article.body-content — BodyContent.vue root; iterates the section's content array of layout items

        • CHROME h2.title — localized "Check Your Understanding" banner
        • CHROME AssessmentsProgress — progress indicator (index/total, stateful)

    • CHROME PortalTarget[name=modal-destination] — portal mount for nested modals (e.g. hero's video modal)

4. RenderNode content dispatch (schema-driven)

Article.vue's own dispatch table only recognizes four section kinds — a strict subset of what a Tutorial (Project) page's Tutorial.vue accepts (which additionally dispatches intro/tasks/resources/volume and does not accept articleBody at all): 

const SectionKind = {
  articleBody: 'articleBody',
  callToAction: 'callToAction',
  hero: 'hero',
  assessments: 'assessments',
};

(Article.vue#L45-L50.) Any section on the wire whose kind is not one of these four fails the component's own prop validator and would not render. Below that section level, the articleBody section's own content array is not block content directly — it is a three-way layout dispatch (columns / contentAndMedia / fullWidth) handled by BodyContent.vue, and only inside a resolved layout item does ordinary block/inline ContentNode dispatch take over (via Article/ContentNode.vue, a thin wrapper around the shared components/ContentNode.vue that force-enables code-listing line numbers and strips heading anchors).
