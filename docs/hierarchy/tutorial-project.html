1. Route and component dispatch

This page is the interactive DocC tutorial: a full RenderNode whose top-level kind discriminator is the wire value "project" (swift-docc's RenderNode.Kind declares case tutorial = "project", line 256 — every layer above the wire calls this a "tutorial", but "project" is what actually lands on disk). Full route/dispatch details and citations are documented in kinds/node-tutorial.html §1; summarized here for this page's own tree.

The route-level dispatcher is views/Topic.vue, which switches on topic.kind via a local TopicKind map ({ article: 'article', tutorial: 'project' }) and, for kind: "project", renders components/Tutorial.vue, passing exactly five props derived from the RenderNode: 

propsFor(topic) {
  const {
    hierarchy, kind, metadata, references, sections, identifier,
  } = topic;
  return {
    [TopicKind.tutorial]: {
      hierarchy, metadata, references, sections, identifierUrl: identifier.url,
    },
    ...
  }[kind];
}

hierarchy and metadata feed the page chrome (section 2); sections is the array this page's own body is built from, each entry re-dispatched by its own kind field through Tutorial.vue's local SectionComponents map (hero → Hero.vue, tasks → SectionList.vue, assessments → Assessments.vue, callToAction → CallToAction.vue); references is the identifier-keyed lookup table those sections resolve topic links, images, code files, and downloads against. This same five-prop hand-off and section dispatch table is what kinds/node-tutorial.html §1 walks through with full line citations — see that page for the RenderNode.Kind/SectionComponents source excerpts verbatim.

2. Page chrome (non-schema)

Tutorial.vue's root template renders exactly one chrome component above the section list: components/Tutorial/NavigationBar.vue, conditionally rendered only when !isTargetIDE (line 14 of Tutorial.vue: <NavigationBar v-if="!isTargetIDE" .../> — Xcode's embedded IDE documentation viewer suppresses this bar entirely).

NavigationBar.vue builds on the shared docc-render/components/NavBase.vue: its template renders <NavBase id="nav" :aria-label="technology" hasSolidBackground>, giving the root chrome element the id #nav and a solid (non-translucent) dark background, distinct from the default, scroll-transparent nav used on documentation pages. Inside NavBase's default slot it places a NavTitleContainer showing the technology name (metadata.category) as the title and a localized "Tutorials" subhead, linked back to the tutorial's technology root via a ReferenceUrlProvider wrapping hierarchy.reference. Inside the tray slot it places, in order:

  • .mobile-dropdown-container > MobileDropdown — a single combined chapter+section picker shown only at small breakpoints (the container is display: none otherwise).
  • .dropdown-container > .primary-dropdown (PrimaryDropdown.vue) — the chapter/tutorial picker, built from chapters (hierarchy.modules) with currentOption set to the page's own title (tutorialTitle, derived from the hero section, not from the RenderNode's own optional title field).
  • ChevronIcon.icon-inline — a static separator glyph between the primary and secondary dropdowns.
  • .secondary-dropdown (SecondaryDropdown.vue) — rendered only v-if="currentSection"; lists the current tutorial's own @Section anchors (optionsForSections, derived from hierarchy.modules[*].projects[*] .sections plus a synthetic "Introduction" first entry) with a live section-progress indicator (sectionIndicatorText, e.g. "Section 2 of 4").

Both PrimaryDropdown.vue and SecondaryDropdown.vue delegate their open/closed toggle behavior to the shared DropdownCustom.vue, and SecondaryDropdown's selection additionally drives a manual scrollToElement mixin call (working around a vue-router scroll-restoration bug) rather than a plain in-page anchor jump. NavigationBar.vue's own scoped CSS layers dark-themed dropdown backgrounds (--color-tutorial-navbar-dropdown-background / -border) on top of the shared NavBase styles, and collapses the desktop grid (.dropdown-container, .separator) in favor of the mobile dropdown at the nav-in-breakpoint mixin's breakpoint.

This page has no sidebar navigator and no search. Unlike the Documentation Topic page (DocumentationTopic.vue, wire kind "symbol"/"article" " — see doc-topic.html for its NavigatorDataProvider/left-rail tree and its Navigator-embedded search field), Tutorial.vue's template contains no navigator sidebar component and no search input at all: navigation within a tutorial project is limited to the Primary/Secondary/Mobile chapter+section dropdowns described above, plus the page's own in-content anchors and the trailing callToAction section's "what's next" link. This is a structural fact of Tutorial.vue's template (no sidebar/search components are imported or rendered anywhere in it), not a configuration flag that could be toggled on.

3. DOM hierarchy: root → chrome → content → leaves

Below is a representative (not exhaustive) walk of the rendered DOM for a tutorial project page, one example subtree per section kind Tutorial.vue actually dispatches on. Tag legend: chrome non-schema UI, node top-level RenderNode kind, section RenderSectionKind, block RenderBlockContent, inline RenderInlineContent, reference RenderReference (from the references map).

  • node div.tutorial — root element for the whole page; wraps the entire TutorialsPageRenderNode (kind "project"). kinds/node-tutorial.html
    • chrome #nav (NavigationBar.vue > NavBase.vue) — technology/chapter/section navigation bar; rendered only when !isTargetIDE; no sidebar, no search (see section 2).
      • chrome .nav-title (NavTitleContainer) — technology name + "Tutorials" subhead, linking to hierarchy.reference.
      • chrome .mobile-dropdown-container > MobileDropdown — combined chapter+section picker (small breakpoint only).
      • chrome .dropdown-container > .primary-dropdown (PrimaryDropdown.vue) — chapter/tutorial picker built from hierarchy.modules.
      • chrome .secondary-dropdown (SecondaryDropdown.vue) — current tutorial's own @Section anchors + progress indicator.
    • chrome main#app-main[tabindex="0"] — wraps the sections array; one child per entry, dispatched by each section's own kind.
      • section .tutorial-hero > div.hero.dark (Hero.vue) — kind: "hero" intro section. kinds/section-hero.html
        • chrome .bg — optional background-image layer, resolved from backgroundImage against the light-variant reference entry in references.
        • block h1.headline > span.eyebrow — Headline component; eyebrow text is the hero's chapter string, title is the hero's own title string (both plain strings on the section, not block/inline content).
        • block .intro (ContentNode.vue dispatch) — renders the hero's content array.
          • block p (paragraph) — kind: "paragraph". kinds/block-paragraph.html
            • inline (bare text node) — type: "text", spliced in as a raw DOM string. kinds/inline-text.html
            • inline code.inline-code — type: "codeVoice", e.g. a symbol name mentioned in the intro copy. kinds/inline-codevoice.html
        • chrome a.call-to-action > PlayIcon + GenericModal > Asset.video-asset — "Watch intro video" trigger, shown only when the hero's video field is set; resolves to a reference video entry. kinds/reference-video.html
        • chrome .metadata (HeroMetadata.vue) — renders estimatedTimeInMinutes, and resolves projectFiles / xcodeRequirement to reference entries.
          • reference (download link) — projectFiles identifier resolved against the references map. kinds/reference-download.html
          • reference (Xcode requirement chip) — xcodeRequirement identifier resolved against the references map. kinds/reference-xcoderequirement.html
      • section .sections > .section (SectionList.vue > Section.vue) — kind: "tasks", one .section per entry in tasks. kinds/section-tasks.html
        • chrome (SectionIntro.vue, via Section.vue's Intro) — renders the task's contentSection intro content/media (non-step body of the @Section/@Steps block).
        • block .steps > .content-container > .step-container > .step (SectionSteps.vue > Step.vue) — one per step entry in stepsSection. kinds/block-step.html
          • block p.step-label — "Step N" label (plain, non-schema chrome text driven by the computed stepNumber).
          • block (ContentNode.vue dispatch over content) — the step's instructional prose.
            • block p (paragraph) kinds/block-paragraph.html
              • inline strong — type: "strong". kinds/inline-strong.html
              • inline a (reference) — type: "reference", an inline link resolved via identifier against references, e.g. to a reference topic entry. kinds/inline-reference.html / kinds/reference-topic.html
          • chrome .asset-container > CodePreview > CodeListing (or Asset for media) — the step's code identifier resolves to a reference file entry rendered by the shared code-listing component; the step's media identifier resolves to an reference image/video entry. kinds/reference-file.html
            • block pre > code (codeListing, via CodeListing.vue) — syntax-highlighted lines of the step's associated code file. kinds/block-codelisting.html
          • block .caption (ContentNode.vue dispatch over caption) — optional step caption content, same block/inline shapes as the main step content.
      • section .assessments-wrapper > .assessments (Assessments.vue) — kind: "assessments", one instance for the section's whole assessments array (quiz questions are stepped through client-side, not all rendered as siblings). kinds/section-assessments.html
        • chrome h2.title — localized "Check Your Understanding" banner (not schema content).
        • chrome AssessmentsProgress — "Question N of M" progress indicator, computed from array length/index, not itself schema content.
        • block .quiz (Quiz.vue) — renders one assessments[activeIndex] entry at a time.
          • block .title (ContentNode.vue over title) — the active question's title block content.
          • block .question-content (ContentNode.vue over content) — the active question's supporting content block content (optional).
          • block fieldset.choices > label — one per entry in choices.
            • block .question (ContentNode.vue over choice.content) — the choice's own label content.
            • block .answer (ContentNode.vue over choice.justification) — revealed only after that choice is checked; the justification/feedback content.
      • section (CallToAction.vue > shared CallToAction.vue "BaseCTA") — kind: "callToAction", DocC's auto-generated "what's next" panel. kinds/section-calltoaction.html
        • chrome (featured media figure) — the section's media identifier resolved to an reference image entry. kinds/reference-image.html
        • block (ContentNode.vue over abstract) — the panel's abstract text.
          • inline (bare text node) — type: "text". kinds/inline-text.html
        • reference a (action link) — the panel's action object, an inline-reference-shaped pointer (type: "reference") to the next tutorial/article, with an optional overridingTitle. kinds/reference-topic.html
    • chrome PortalTarget[name="modal-destination"] — portal-vue teleport target for modals raised from within tutorial content (e.g. the Hero's video modal above).

Not walked above for brevity, but reachable the same way: orderedList/ unorderedList/aside/table/row blocks and emphasis/image/newTerm inline runs anywhere ContentNode.vue dispatches over block/inline content (hero intro, step content/ caption, quiz title/content/choices, callToAction abstract) — see section 4 for the full reachable set specific to this page.

4. RenderNode content dispatch (schema-driven)

Tutorial.vue's SectionComponents map (see section 1) is a closed set of exactly four section kinds. Two other section kinds that share the same top-level RenderSection oneOf (volume, resources) exist on the schema but belong to the Tutorial Overview page (kind "overview"), not this page — Tutorial.vue has no entry for them and would silently drop a section with either kind.

Section kindComponentKind page
heroTutorial/Hero.vuesection-hero.html

assessmentsTutorial/Assessments.vuesection-assessments.html

Underneath those four sections, block and inline content is dispatched generically by the shared ContentNode.vue (its BlockType/InlineType maps), except for step, which SectionSteps.vue intercepts and routes to its own dedicated Step.vue instead of letting ContentNode.vue render it generically (see block-step.html).
